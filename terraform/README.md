**Author:** Vinay Kumar Kondoju

## Modules Reference

### `vpc`
Provisions a VPC with configurable CIDR, public and private subnets across multiple AZs, a NAT gateway in each public subnet (single NAT for dev to reduce cost), an internet gateway, and all required route tables.

### `eks`
Creates an EKS cluster with a managed node group. Enables the OIDC provider (required for IRSA). Outputs the cluster endpoint, CA data, and OIDC provider ARN consumed by the `iam` module.

Node group defaults (dev): `min=1`, `desired=2`, `max=3`, instance type `t3.medium`.

### `ecr`
Creates ECR repositories with image scanning on push and `prevent_destroy = true` lifecycle protection so images are not accidentally deleted by a `terraform destroy`.

### `kms`
Creates a single symmetric CMK with automatic key rotation enabled. Used for DynamoDB, S3, and Secrets Manager encryption. Key ARN is passed into every other module that needs encryption.

### `dynamodb`
Accepts a map of table names to GSI configurations. Creates all 19 tables with:
- Partition key `id` (String)
- Per-table GSIs using the provided attribute names
- SSE enabled with the KMS CMK
- Pay-per-request billing (no capacity planning needed)

### `cognito`
User Pool with password policy, MFA optional, email verification. App Client configured for SRP (Secure Remote Password) auth flow. Outputs `user_pool_id` and `client_id` injected into Secrets Manager by the `secrets-manager` module.

### `secrets-manager`
Creates one secret per service, keyed by service name. The secret value is a JSON object containing all environment variables the service needs at runtime:

```
AWS_REGION, AWS_S3_BUCKET, AWS_KMS_KEY_ID,
JWT_SECRET, JWT_REFRESH_SECRET, JWT_EXPIRY, JWT_REFRESH_EXPIRY,
CORS_ORIGIN,
AWS_BEDROCK_REGION, AWS_BEDROCK_TEXT_MODEL_ID, AWS_BEDROCK_EMBEDDING_MODEL_ID,
COGNITO_USER_POOL_ID, COGNITO_CLIENT_ID,
DYNAMODB_TABLES (JSON map of table names),
S3_BUCKET_NAME
```

`JWT_SECRET` and `JWT_REFRESH_SECRET` are generated by `random_password` resources ‚Äî they are created once and stored in the Terraform state, never in version control.

### `iam`
Creates IRSA (IAM Roles for Service Accounts) ‚Äî one role per service. Each role:
- Trusts the EKS OIDC provider scoped to the specific Kubernetes service account
- Has an inline policy granting access to: Secrets Manager (own secret only), DynamoDB (all 19 tables), S3 (documents bucket), KMS (decrypt only), Bedrock (ai-service only, scoped to specific model IDs), CloudWatch Logs

Also creates the `AWSLoadBalancerControllerIAMPolicy` role used by the in-cluster AWS LBC.

### `kms`, `s3`, `acm`, `route53`, `waf`, `sns`, `cloudwatch`
Standard module wrappers around their respective AWS resources, accepting environment name and common tags as inputs.

## Remote State

Terraform state is stored remotely in S3 with DynamoDB locking:

| Setting | Value |
|---------|-------|
| S3 Bucket | `procurement-tf-state-global` |
| State Key | `environments/{shared,dev,prod}/terraform.tfstate` |
| DynamoDB Lock Table | `terraform-state-lock` |
| Region | `us-east-1` |

Each environment's `backend.tf` configures these values. The state bucket itself is managed outside Terraform (created manually before first apply) to avoid the bootstrap chicken-and-egg problem.

## GitHub Actions Workflows

### Applying Infrastructure

```
GitHub ‚Üí Actions ‚Üí Terraform Apply ‚Üí select environment: shared | dev | prod
```

The `terraform-apply.yml` workflow:
1. Authenticates to AWS using GitHub OIDC (no access keys)
2. Runs `terraform init` (pulls state from S3)
3. Runs `terraform plan -out=tfplan`
4. Runs `terraform apply -auto-approve tfplan`
5. Prints all outputs

### Destroying Infrastructure

```
GitHub ‚Üí Actions ‚Üí Terraform Destroy ‚Üí type environment name to confirm
```

The `terraform-destroy.yml` workflow requires typing the environment name as confirmation before destroying. It will never destroy the `shared` environment automatically.

### CI Validation (on every PR)

The `ci.yml` workflow runs:
- `terraform fmt -check -recursive terraform` ‚Äî enforces consistent formatting
- `terraform init -backend=false && terraform validate` ‚Äî syntax and schema check for all three environments

## Applying Manually (Local)

```bash
# Prerequisites: AWS credentials configured, Terraform >= 1.5

# 1. Apply shared resources first (ECR, Route53, ACM)
cd terraform/environments/shared
terraform init
terraform apply

# 2. Apply dev environment
cd terraform/environments/dev
terraform init
terraform apply

# 3. After EKS is up, bootstrap the cluster (ArgoCD + LBC)
bash scripts/bootstrap-cluster.sh dev
```

## Key Design Decisions

**No shared IAM credentials between services.** Every service has its own IRSA role and its own Secrets Manager secret. Compromise of one service cannot escalate to others.

**Secrets Manager as the single source of truth for runtime config.** All environment-specific values (Cognito IDs, table names, JWT secrets, Bedrock model IDs) are injected via Secrets Manager at pod startup ‚Äî never baked into Docker images or Helm values.

**OIDC instead of access keys.** GitHub Actions authenticates to AWS via the OIDC federation endpoint. The role assumption is constrained to the specific GitHub org/repo/branch, so no long-lived credentials exist anywhere.

**`prevent_destroy` on ECR.** Container images are expensive to rebuild and represent proven, scanned builds. ECR repos are protected with `lifecycle { prevent_destroy = true }` so a mistaken `terraform destroy` cannot wipe them.
